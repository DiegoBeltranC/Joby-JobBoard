5. Conclusiones — ¿Qué arquitectura elegiría?

Análisis basado en el proyecto JobBy: escalabilidad, equipo, mantenibilidad y presupuesto.

🥇 Primera elección
Arquitectura en Capas — Layered Architecture

Para el proyecto JobBy, elegiría la Arquitectura en Capas como primera opción. Esta arquitectura organiza el sistema en cuatro capas bien definidas (Presentación → Lógica de Negocio → Acceso a Datos → Base de Datos), lo que se alinea perfectamente con el stack elegido: Next.js 15 con Server Actions actúa como capa de presentación y negocio, Prisma ORM como capa de acceso a datos y PostgreSQL como capa de persistencia.

Esta separación de responsabilidades hace que cada parte del sistema sea fácil de entender, depurar y modificar de forma independiente. Si se necesita cambiar la lógica de vacantes, solo se toca vacantes.ts sin afectar la UI ni la base de datos. Es la arquitectura que ya implementamos de forma natural en el proyecto, lo que significa que no hay costo de curva de aprendizaje adicional.

▲ Escalabilidad deseada
Escala verticalmente de forma sencilla. Para un proyecto universitario con crecimiento moderado es suficiente; en el futuro puede evolucionar a microservicios extrayendo capas.
▲ Tamaño del equipo
Ideal para equipos pequeños (2–5 personas). Las capas actúan como fronteras naturales de trabajo: un miembro trabaja en UI, otro en Server Actions, otro en el schema de BD.
▲ Mantenibilidad y modularidad
Alta mantenibilidad: cada capa tiene una única responsabilidad. Agregar una nueva feature solo requiere tocar el archivo de acción correspondiente y la página asociada.
▲ Presupuesto y tiempo
Costo mínimo: un solo deploy en Vercel (gratis). Tiempo de desarrollo corto porque Next.js ya impone esta estructura. No se necesita infraestructura adicional.
🥈 Segunda elección
Arquitectura Cliente-Servidor — Client-Server

Como segunda opción elegiría la Arquitectura Cliente-Servidor. En este modelo existen dos actores bien diferenciados: el cliente (el navegador del usuario) que hace solicitudes, y el servidor (Next.js corriendo en un VPS o en Vercel) que procesa esas solicitudes y responde con datos o páginas completas. La comunicación entre ambos ocurre siempre a través de HTTP, con una frontera clara de quién pide y quién responde.

En JobBy esta arquitectura se ve de forma muy concreta: cuando un estudiante abre /dashboard, el navegador le hace un HTTP GET al servidor → Next.js ejecuta el React Server Component en el servidor, consulta PostgreSQL a través de Prisma y devuelve el HTML ya armado al cliente. No hay API REST intermedia, no hay JSON de ida y vuelta: el servidor entrega la página lista para renderizar. Cuando el estudiante envía un formulario (postular a una vacante, editar su perfil), el navegador dispara un Server Action POST → el servidor valida, escribe en BD y responde con el nuevo estado. El cliente nunca toca la base de datos directamente; toda la lógica sensible vive y muere en el servidor.

Esto la hace la segunda arquitectura más adecuada porque Next.js ya implementa este modelo de forma nativa: el Middleware de autenticación actúa como guardián del servidor, el App Router decide qué componente ejecutar en el servidor y cuáles en el cliente, y las cookies de sesión nunca abandonan el lado del servidor. A diferencia de microservicios o SOA, no se necesita coordinar múltiples procesos, gestionar contratos entre APIs ni montar infraestructura adicional. Un único servidor resuelve todo, reduciendo la complejidad operativa a casi cero.

◆ Escalabilidad deseada
Escala horizontalmente levantando más instancias del servidor Next.js detrás de un balanceador de carga. Vercel hace esto automáticamente con Edge Functions y serverless: cada ruta puede escalar de forma independiente sin tocar el código. Para la carga esperada de una bolsa universitaria es más que suficiente. La única limitación aparece si el tráfico crece exponencialmente, caso en el que habría que separar el servidor de BD en su propia instancia — algo trivial sin reescribir lógica.
▲ Tamaño del equipo
Ideal para equipos pequeños (2–4 personas). La división cliente/servidor es el vocabulario base de cualquier desarrollador web: todos saben qué va en el navegador y qué en el servidor. No se necesitan especialistas en DevOps, orquestación de contenedores ni diseño de contratos entre APIs. Un solo miembro puede arrancar el proyecto completo con npm run dev y trabajar en ambos lados de la frontera sin fricción.
◆ Mantenibilidad y modularidad
La mantenibilidad es buena siempre que se mantenga disciplina interna: si el código del servidor crece sin organización, puede convertirse en un monolito difícil de navegar. Por eso la combinamos con la Arquitectura en Capas: las capas imponen el orden interno (cada Server Action en su dominio, Prisma solo en la capa de acceso a datos) mientras que cliente-servidor define la frontera externa. Juntas resuelven tanto la organización como la separación de responsabilidades.
▲ Presupuesto y tiempo
Costo de infraestructura: $0 con Vercel plan gratuito. Tiempo de implementación: prácticamente nulo porque Next.js 15 impone esta arquitectura por diseño — no hay que tomar decisiones sobre cómo estructurar el cliente y el servidor, el framework ya lo resuelve con el App Router, los Server Components y los Server Actions. Esto libera al equipo para enfocarse en la lógica de negocio en lugar de la infraestructura.
Reflexión final: Ambas arquitecturas no son excluyentes — en la práctica, JobBy las combina: usa la separación Cliente-Servidor que provee Next.js y, dentro del servidor, organiza el código en Capas. Esta combinación ofrece lo mejor de los dos modelos: simplicidad operativa del cliente-servidor y claridad de responsabilidades de las capas. Para un proyecto universitario con equipo reducido, presupuesto limitado y plazos de entrega cortos, esta combinación es la decisión técnica más sólida y pragmática.
